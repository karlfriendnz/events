### Activity `booking_flow` — wizard vs scheduler vs item
Each activity row has a `booking_flow` column:

End_at is computed by adding `period_count × N` of `period_unit` to start. `term_type='recurring'` rentals write `bookings.is_recurring=true` (renewal cron is phase 2 — for now they live as open-ended single bookings ending at the first period's end). Resource picker honours `activities.assignment_mode` (`'system'` auto-assigns, `'member'` forces explicit pick, `'either'` shows "Assign me one" + the unit list).

Set per-activity in the activity detail page. Both `pages/bookings/new.vue` and `pages/book/index.vue` dispatch on `booking_flow` to render the right component.

## Key Conventions

### DB Access
Always use `useDb()` — never `useSupabaseClient()` directly in pages/components.
```ts
const db = useDb()
const { data, error } = await (db.from as any)('table_name').select('*').eq('org_id', orgId.value)
```
The `(db.from as any)` cast bypasses stale generated types — intentional.

### Org Context
```ts
const { orgId } = useOrg()   // always scope queries to orgId.value
```

### Card surface
`class="card"` = the standard white card (`bg-white` + `gray-200` border + `rounded-xl`), defined in `assets/css/main.css`. Pair with `p-*` / `overflow-*` utilities. Equivalent to `<AppCard>`'s frame but without the title/header slots — use `<AppCard>` when you want the titled header, plain `class="card"` for a bare surface.

### Primary Brand Color
`#1E2157` → use Tailwind token: `bg-primary`, `text-primary`, `hover:bg-primary-hover`.
For PrimeVue Buttons (can't use Tailwind tokens there): `style="background:#1E2157;border-color:#1E2157"`.

## Mobile-First Design System — SYSTEM RULES (apply to EVERY page)

The whole app is **mobile-first, like a mobile app**. The canonical reference design is **`pages/people/[id].vue`** (the People profile) — copy its shell, padding, tabs, and form patterns. Build/verify at **390px** (mobile), 768px (tablet), 1280px (desktop). Breakpoint = Tailwind `sm:` (640) / `md:` (768).

0. **Type scale** (use these — do NOT invent `text-[10px]`/`text-[11px]`/`text-[9px]` arbitrary sizes): page title `text-lg sm:text-2xl font-semibold`; card/section title `text-sm font-semibold` (what `<AppCard>` uses); body / field values `text-sm`; labels, meta, overlines, badges, column headers `text-xs` (overlines add `font-bold uppercase tracking-wide`); the smallest you go is `text-xs`. Stat/number tiles `text-lg font-bold`. (Icon-only `<i>` sizes are exempt.) Normalised across `pages/people/[id].vue`, `<PersonCirclesEditor>`, `<PersonFormBuilder>`; the rest of the app is a follow-up sweep.
   - **MultiSelect chips are uniform system-wide** (`text-xs`/12px, soft slate fill, small remove icon). This is enforced by a **global rule** in `assets/css/main.css` (`.p-multiselect-chip-item .p-chip`) covering EVERY `display="chip"` MultiSelect (User Role, Communication, member groups, event pickers, …), and `<ChipMultiSelect>`'s custom `.chip-ms` is tuned to match. **Never hand-size chips per-instance** — they must all look identical. Use `<ChipMultiSelect>` only when you also need the "+N more" single-line overflow.
1. **Page padding** — every page root (or its content container) uses `p-3 sm:p-6` (12px mobile → 24px desktop). NEVER a bare `px-6`/`p-6` that stays 24px on mobile. Inner row padding: `px-4 sm:px-6`. Section gaps: `space-y-4 sm:space-y-5`. Consistent edge inset across all pages is a hard rule.
2. **Detail-page shell** — follow the profile: a `card p-0 overflow-hidden mb-4` header holding a top row (`px-4 sm:px-5`: back-chevron + optional avatar + `<h1>` title/subtitle on the left, Edit/Save + `⋯` `<Menu>` actions on the right) then a tab strip. Title is **`text-lg sm:text-2xl`** — never `text-2xl`/`text-3xl` on mobile (too big). Subtitles `text-xs`.
3. **Tabs** — underline strip: `flex … overflow-x-auto overflow-y-hidden no-scrollbar`, each tab `border-b-2 -mb-px whitespace-nowrap`, active = `border-primary text-primary`, idle = `border-transparent text-gray-500`. For pages with **many tabs** (e.g. `events/[id]`), use a **`<Select>` dropdown on mobile (`md:hidden`)** bound to `activeTab` + the underline strip on desktop (`hidden md:flex`). Do NOT use wrapping pill buttons.
4. **Forms / detail rows** — label **above** the value on mobile, side-by-side on desktop: `flex flex-col sm:flex-row sm:items-center gap-1 sm:gap-6`, label `w-full sm:w-28`. Prefer `<SettingsRow>` (already does this). Never a fixed `w-28` label glued left to the value on mobile (causes ugly value wrapping). Hover-only edit pencils get `hidden sm:block` (tapping the row opens edit on mobile).
5. **Tables → card lists on mobile** — `md:hidden` stack of tappable cards + `hidden md:block overflow-x-auto` DataTable. Calendars → agenda/list on mobile. Multi-column filter tab strips → a `<Select>` on mobile.
6. **Dialogs** — `:style="{ width: '95vw', maxWidth: '<n>px' }"` so modals never overflow the viewport.
7. **Buttons** — full-width on mobile where they're the row's primary action (`w-full sm:w-auto justify-center`); icon-only (drop the label) when space is tight. Touch targets ≥44px.
8. **Shell** — left icon rail is `hidden md:flex`; mobile uses the bottom tab bar + "More" sheet in `layouts/default.vue`. `<main>` gets `pb-16 md:pb-0` to clear the bottom bar.

Leave the **form builders** (`FormBuilder`, events forms-tab field editor, scheduler/calendar grids, venue sub-venue map, drag-reorder pages) out of the mobile pass for now — flagged as hard-to-mobilise.

### Advanced mobile patterns (learned — apply when polishing a screen)
9. **Master-detail (list + editor) on mobile** — do NOT show a tall list strip beside/above the editor. Replace the list with a **`<Select>` dropdown chooser** (`lg:hidden`, bound to the selected item, `@update:modelValue` calls the same select handler) + an inline Add/action button; hide the desktop list panel (`hidden lg:flex`). The editor then gets the full height. Reference: the Sessions tab in `pages/events/[id].vue` (session chooser) + `<SessionEditor>`.
10. **Scroller height caps go on the scroller, not the panel** — a `max-h-[Nvh]` on a flex *panel* whose child is `flex-1 overflow-y-auto` does NOT give the child a definite height, so the list under-fills and leaves a dead gap. Put `max-h-[34vh] lg:max-h-none` directly on the `overflow-y-auto` element (and give it `lg:flex-1` for desktop).
11. **Tab/pill rows scroll, never wrap** — a row of pills should be one horizontal strip: `flex … overflow-x-auto no-scrollbar`, items `shrink-0 whitespace-nowrap`. Wrapping pills into 2–3 rows looks broken. **Pin the row's primary action** (e.g. Take Attendance) in a `shrink-0` cell beside the scroller, icon-only on mobile via `:pt="{ label: { class: 'hidden sm:inline' } }"`.
12. **Field rows: label-row above, full-width input below** — for an input that also carries side metadata (badges like "9 inheriting", lock/from-master actions), use a header row `flex items-center justify-between` (label left, badge/action cluster right) **above** a full-width input — never float the metadata chip beside a shrunken input. Reference: `<SessionEditor>` Session Info.
13. **Many-tab page header** — `<Select>` dropdown on mobile (`md:hidden`) + an underline strip on desktop (`hidden md:flex`); the strip left-aligns (`justify-start xl:justify-center`) so the first tab never clips when tabs overflow. Reference: `pages/events/[id].vue` top bar.
14. **PrimeIcons `.pi` beats `.hidden`** — to hide a `<i class="pi …">` responsively you MUST use the important modifier: `!hidden sm:!block` (plain `hidden` loses to PrimeIcons' `display:inline-block`).
15. **grid-layout-plus insets** — KEEP the negative side-margins (`marginLeft/Right: -18px`) at ALL widths; they cancel the library's own outer margin so widgets align to the page padding. Removing them on mobile *doubles* the inset (~30px). Mobile stacking is handled by the `displayLayout` computed (force `x/y/w`), not by dropping the margin.
16. **Row actions stay light** — a stat/summary row's action (e.g. "Take attendance" on the event Overview Attendees row) is a subtle `text`-style link, not a big solid button. Reserve full-width solid buttons for a step/section's single primary CTA.

**Status:** an app-wide responsive pass has been completed across all areas (events, bookables/activities, booking flows, settings/admin, people/groups/dashboard, reporting/finance/public). Patterns applied everywhere: page/section padding `p-3 sm:p-6`; fixed-width `<Dialog>` → `:style="{ width:'95vw', maxWidth:'NNN' }"`; master-detail `grid-cols-[Npx_1fr]` → `grid-cols-1 md:grid-cols-[Npx_1fr]`; un-prefixed `grid-cols-2/3/4` → `grid-cols-1 sm:grid-cols-N`; `<table>`/`<DataTable>` wrapped in `overflow-x-auto` (or `md:hidden` card lists for the heaviest, e.g. `FeeLineItemsTable`); detail rows stack (`flex-col sm:flex-row`, `w-full sm:w-NN` labels, hover pencils `!hidden sm:!block`); fixed-height shells use `h-[calc(100vh-3.5rem-4rem)] md:h-[calc(100vh-3.5rem)]` (subtract the mobile bottom nav to avoid a 2nd scrollbar); grid-layout-plus dashboards KEEP their negative side-margins at all widths (see advanced rule 15). NOTE: `<SettingsRow>` now switches stack→row at **`sm:`** with a **`w-28`** label (was `md:`/`w-36`) to match the `/people/[id]` canonical reference — affects every settings/detail row app-wide.

## Configuration / slot model (Phase 2 booking integrity)

Configurations are **named groups of slots**, where each slot lists physical sub-venues that get booked atomically together. So a Tennis Court that needs both halves and quarters has:
- Physical sub-venues: Q1, Q2, Q3, Q4 (the **finest** division — only level that exists as `bookables` rows)
- "Quarters" config: 4 slots, each one quarter
- "Halves" config: 2 slots — Half A = {Q1, Q2}, Half B = {Q3, Q4}

When a booking targets a slot, `<BookingScheduler>` inserts:
- One **primary** booking on the first member sub-venue (no `parent_booking_id`)
- One **child** booking per remaining member, each with `parent_booking_id` = primary's id

So booking "Half A" on Court 1 writes two `bookings` rows on Q1 and Q2 — both calendars show busy, no double-booking possible. The pre-flight check in `submit()` queries every member sub-venue across the picked slots before writing, aborting with a toast if any overlap exists.

**Known gap:** the calendar grid (`<BookingsCalendar>` / `<SubVenueScheduler>`) doesn't yet visually cross-block sub-venues sharing a slot — Q1's column shows free even when Q2 is booked in the same Halves slot. The pre-flight check stops the bad write; the visual polish is still to come.

## Access control flow

End-to-end: org-wide door/light catalogue → per-venue connection → booking-side code + unlock window.

1. **Catalogue** (`<AccessControlList>` at `/bookables?tab=access`) — staff CRUD `doors` and `light_zones` rows. Hardware `provider` + `id` are free text for now (real vendor integration is later).
2. **Connection** (`<BookableAccessEditor>` on `/bookables/:id?tab=access`) — staff toggle `bookables.access_enabled` and pick which doors / light zones serve this venue; saved into `bookable_doors` and `bookable_light_zones`.
3. **Booking submit** — every booking insert path calls `$fetch('/api/finalize-access', { body: { bookingId } })` after the row exists, **only when the booking is CONFIRMED** (PENDING bookings don't generate codes — no point until approved). Wired in:
   - `server/api/public-booking.post.ts` (after notification + confirmation email)
   - `components/BookingScheduler.vue` (one call per primary booking; child slot members inherit)
   - `components/BookingWizard.vue`
   - `components/ItemBooker.vue`
4. **`/api/finalize-access`** — idempotent. Loads the booking + bookable, no-ops when access isn't enabled / already finalised / the booking is a child of a multi-bookable slot. Otherwise: generates an N-digit numeric code (length from `bookables.access_code_length`), updates `bookings.access_code`, materialises one `physical_schedules` row per connected door (with the unlock-before/after window) and one per light zone (with the lighting-ramp window + `level_percent`). Then if `bookables.access_code_delivery` is `email` or `both`, calls `/api/send-access-code-email` and stamps `bookings.access_code_delivered_at`.
5. **`/api/send-access-code-email`** — Resend-backed email containing the code, the venue, the unlock window message. Same stub-when-no-RESEND_KEY pattern as the other senders. SMS path is intentionally a stub for now.

## Registration submission flow

The reusable registration engine — one form definition, many contexts. Built to back event registration, group registration, competition entry, website enquiries, etc.

1. **Design** — staff build the form in the events forms tab (`<FormDesigner>`), persisted to `registration_forms.config` (subjects in `config.groupProfiles`, fields in `config.groupFields`).
2. **Public page** — `pages/r/:context/:id` (embed layout, guest-accessible) loads the context entity, its linked form config, org theme, and (events) sessions + `fee_components`, then renders `<FormRenderer>`.
3. **Fill + submit** — `<FormRenderer>` renders real inputs for every subject/instance/field (respecting min/max counts, visibility conditions, single-page vs step wizard, payment + terms), validates, and POSTs a normalised payload to `/api/public-form-submit`.
4. **`/api/public-form-submit`** (service-role, like `public-booking`) — derives `org_id` **server-side** from the context entity (never trusts the client), then:
   - **Materialises persons** for each person-kind subject instance (match by email else create), mapping core fields → person columns (`first_name`/`last_name`/`email`/`phone`/`dob`/`gender`) and other answers → `persons.custom_fields` **keyed by `field_definitions.id`** (resolved via the `org_ancestors` RPC; falls back to label keys when no definition matches — this is how national/inherited answers surface on the member profile).
   - **Event context**: writes a `registrations` row (`form_answers` = full payload, `status`, `total_amount`), upserts `invitees` per person (roster/attendance light up), and `registration_sessions` for selected sessions.
   - **Group context**: upserts `member_group_memberships`.
   - **Always**: writes one `form_submissions` row (uniform record) + a `registration.created` staff notification (→ `send-notification-email`).
   - Returns `{ success, submissionId, registrationId, personCount }`.
5. **Known follow-ups**: discount engine + payment *processing* (only method captured today), customer confirmation email (staff notification only so far), and a staff "copy/share registration link" button. Field `connected_to:'event'` answers are currently also written to the person (minor).
